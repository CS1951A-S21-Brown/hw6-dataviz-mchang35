Answers to Written Questions

  1. The first graph, titled, "Top 20 Genres on Netflix," answers the first question. It is a bar graph, with the bars growing horizontally to the right and exact counts of titles at the end of each bar. This bar is dynamic because users can select the time range that this graph can cover (the default is the entire range covered by the dataset, i.e., 1925-2020).

    The second graph, titled, "Average Runtime of Movies by Release Year (1975-2020)," answers the second question. It is a scatter plot with a tooltip, which highlights the current point the user's mouse is hovering over, as well as information about that point, such as the release year that point corresponds to and the average runtime of movies for that release year. This graph only considers the years 1975-2020 because I wanted to only include years that had five or more movies; most years before 1975 had less than five movies and most years after 1975 had at least five movies.

    The third graph answers the third question. This graph is a network graph, where each node represents an actor and each link represents at least one movie/TV show that two actors (whose nodes the link connects) have in common. This graph is dynamic, as it can change based on the criteria, which determines which actors are included in the graph and which are not. The first criteria is the default, which is that actor pairs are only included if both actors star together in at least six movies/TV shows. The second option for criteria is that an actor is only included if, individually, he/she has acted in movies/TV shows alongside at least 125 other actors. This graph represents data for the entire dataset (from the years 1925-2020). This graph also has a tooltip, such that one can see a node/actor's or link's information by hovering over it.

Another reason why D3 was helpful in creating and improving my visualization was that it made it easy to implement interactive elements, such as the customizable time range for the first graph and the criteria options for the third graph. It was easy to switch up certain parts of the graph or apply a slightly different set of manipulated data to a graph in order to reflect the customization that the user requested for. In a way, this third reason can be interpreted as an extension of the first. Additionally, it was helpful to be able to look at the HTML file and see how it translated to the .js file and vice versa. In all, D3 does a great job of easily binding data points to graphical elements and makes it easy to visualize the same set of data in many different ways/with different filters and specifications.

Answers to Accessible Data Visualization Questions

  1. I tried to make my dashboard accessible to those who are color blind by making the colors distinct. I also wanted to make sure that, when hovering over the points in the scatter plot or the nodes and links in the network graph, that more would change than just the color; in both graphs, the item over which the user's mouse is hovering also changes size such that it is slightly bigger (for the links, their stroke becomes thicker). This makes it easier for those with color blindness to see the exact element they are hovering over and seeing information for. Additionally, for the links in the network graph, I tried to make it such that the color changed pretty clearly as well when a user hovered over it. Furthermore, nevertheless, I tried to use a palette that was colorblind-friendly, with darker and more distinct hues. This is related to the Medium article's discussion of being mindful of those with colorblindness. Specifically their example of changing the thickness of a line is something I incorporated into my network graph with the links.

    My dashboard may not be as accessible to those who are blind, as I did not make any of my data able to be heard, connecting the data information or visualization information to an audio format. The Medium article mentions allowing users to tab through each data point, as well as sonification. My dashboard also may not be as accessible to those with low vision. Specifically, the network graph, for some filters/selections, can be cluttered and it can be very difficult to tell what links connect which nodes.

    Some actions I can take to make my dashboard more accessible to those who are blind is adding some sort of sonification or audible element so that that audience can better understand the data or at least what the graphs are conveying. I can also work on the network graph to make it less cluttered by refining the forceSimulation's attributes so that nodes are more evenly spaced within the graph's bounds and it is easier to see links.

  2. The primary factor that kept me from taking these steps (and also from taking steps that would have made my visualizations more visually appealing) was that it was difficult to learn how to do them correctly. I had a lot of trouble finding good documentation about some features, so I ended up not including them. For instance, I worked for hours trying to figure out how to construct just the right forceSimulation that allowed users to drag nodes but also would reposition nodes smoothly as users dragged nodes so that users could make the nodes and their links easier to see. I did trial and error with gravity, distance of links, strength, charge, alpha, and more attributes, and in the end, I didn't have enough time to make the force such that it would be more accessible to those with low vision. I also struggled with creating a range slider for the first graph, so that it would be easier for users to select a range of years. This also was due to limited time and very limited expertise. I sought many sources to figure this out, but I could never quite get it right.
